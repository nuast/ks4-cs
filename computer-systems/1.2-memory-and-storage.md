# 1.2 Memory and Storage

> **Component:** J277/01 – Computer Systems | **Section:** 1.2 | **Exam:** J277/01

---

## What you need to know

This topic covers two big ideas: **how data is stored** (primary and secondary storage) and **how data is represented** in binary. You need to be able to convert between number systems, calculate file sizes, and understand how different types of data (text, images, sound) are stored and compressed.

## Key terms

| Term | Definition |
|------|-----------|
| RAM | Random Access Memory – volatile primary storage; holds data and programs currently in use |
| ROM | Read-Only Memory – non-volatile; stores the boot program (BIOS/firmware) |
| Volatile | Data is lost when power is removed |
| Non-volatile | Data is retained when power is removed |
| Virtual memory | Part of secondary storage used as an extension of RAM when RAM is full |
| Bit | The smallest unit of data (0 or 1) |
| Byte | 8 bits |
| Nibble | 4 bits |
| ASCII | American Standard Code for Information Interchange – 7-bit character encoding (128 characters) |
| Unicode | Extended character encoding standard (up to 32-bit) supporting most world languages and symbols |
| Pixel | The smallest element of a digital image |
| Colour depth | Number of bits used to represent each pixel |
| Resolution | Number of pixels in an image (width × height) |
| Sample rate | Number of sound samples captured per second (Hz or kHz) |
| Bit depth | Number of bits used to record each sound sample |
| Lossy compression | Removes some data permanently; smaller file, lower quality |
| Lossless compression | Reduces file size without losing any data; original can be restored exactly |

## Core notes

### Primary storage: RAM and ROM

| | RAM | ROM |
|--|-----|-----|
| Volatile? | Yes – data lost when powered off | No – data retained |
| Contents | Currently running programs and data | Boot program / firmware |
| Writable? | Yes | No (read-only) |
| Speed | Fast | Fast |

- **RAM** is used by the CPU to store the operating system, running applications, and current data.
- **ROM** stores the **BIOS** (Basic Input/Output System), which runs when the computer first powers on.

### Virtual memory
- When RAM is full, the OS moves unused data from RAM to a section of secondary storage (the hard drive) called **virtual memory**.
- Allows more programs to run simultaneously than RAM alone would allow.
- **Disadvantage:** secondary storage is much slower than RAM, so performance drops when virtual memory is heavily used (thrashing).

### Secondary storage comparison

| Type | Examples | Speed | Capacity | Cost per GB | Durability |
|------|----------|-------|----------|-------------|------------|
| **Magnetic** | HDD | Medium | Very high | Low | Fragile (moving parts) |
| **Optical** | CD, DVD, Blu-ray | Slow | Low–medium | Low | Scratches, degrades |
| **Solid state** | SSD, USB flash drive | Fast | High | Higher | Robust (no moving parts) |

### Data units

| Unit | Size |
|------|------|
| 1 nibble | 4 bits |
| 1 byte | 8 bits |
| 1 kilobyte (KB) | 1,000 bytes |
| 1 megabyte (MB) | 1,000 KB |
| 1 gigabyte (GB) | 1,000 MB |
| 1 terabyte (TB) | 1,000 GB |
| 1 petabyte (PB) | 1,000 TB |

> OCR uses decimal prefixes (1 KB = 1,000 bytes) in its specification.

### Number systems

**Binary (base 2) to denary (base 10)**

| 128 | 64 | 32 | 16 | 8 | 4 | 2 | 1 |
|-----|----|----|----|---|---|---|---|
| 1   | 0  | 1  | 1  | 0 | 0 | 1 | 0 |

`1×128 + 0×64 + 1×32 + 1×16 + 0×8 + 0×4 + 1×2 + 0×1 = 178`

**Denary to binary** – repeatedly divide by 2 and record remainders (read bottom to top).

**Hexadecimal (base 16)** uses digits 0–9 and A–F (A=10, B=11, C=12, D=13, E=14, F=15).

Converting hex to binary: convert each hex digit to 4 bits.  
Example: `2F` → `0010 1111`

Converting binary to hex: split into 4-bit groups, convert each.  
Example: `10111001` → `1011 1001` → `B9`

### Binary addition and overflow

```
  0110 1001
+ 0101 1110
-----------
  1100 0111   ← result fits in 8 bits (no overflow)

  1100 0111
+ 0110 1010
-----------
1 0011 0001   ← 9th bit needed = OVERFLOW (result too large for 8 bits)
```

**Overflow** occurs when the result of a binary operation is too large to be stored in the available number of bits.

### Character encoding

- **ASCII:** 7-bit code → 128 characters (letters, digits, punctuation, control characters). Extended ASCII uses 8 bits (256 characters).
- **Unicode:** up to 32-bit → covers most world languages, emojis, special symbols.
- **Why Unicode?** ASCII only covers the Latin alphabet; Unicode allows computing in any language.

### Image file size

```
File size (bits) = image width (px) × image height (px) × colour depth (bits)
```

Example: 1920 × 1080 image, 24-bit colour depth  
`1920 × 1080 × 24 = 49,766,400 bits = 6,220,800 bytes ≈ 6.22 MB`

### Sound file size

```
File size (bits) = sample rate (Hz) × bit depth (bits) × duration (s) × channels
```

Example: 44,100 Hz sample rate, 16-bit depth, 3 minutes (180 s), stereo (2 channels)  
`44,100 × 16 × 180 × 2 = 254,016,000 bits = 31,752,000 bytes ≈ 31.75 MB`

### Compression

| | Lossy | Lossless |
|--|-------|---------|
| Data lost? | Yes – permanently | No |
| File size | Smaller | Larger than lossy |
| Quality | Reduced | Original quality preserved |
| Use cases | MP3, JPEG, MP4 | PNG, ZIP, FLAC, GIF |
| Reversible? | No | Yes |

- **Lossy** is suitable when small file size is more important than perfect quality (e.g. streaming music, web images).
- **Lossless** is essential where data must be exact (e.g. text documents, medical images, software files).

## Worked examples

**Example 1 – Binary to denary conversion**

Convert `01101101` to denary:

| 128 | 64 | 32 | 16 | 8 | 4 | 2 | 1 |
|-----|----|----|----|---|---|---|---|
| 0   | 1  | 1  | 0  | 1 | 1 | 0 | 1 |

`64 + 32 + 8 + 4 + 1 = 109`

---

**Example 2 – Image file size calculation**

A 640 × 480 pixel image uses 8-bit colour depth.

```
File size = 640 × 480 × 8 = 2,457,600 bits
         = 2,457,600 ÷ 8 = 307,200 bytes
         = 307,200 ÷ 1000 = 307.2 KB
```

---

**Example 3 – Hex conversion**

Convert `3A` to binary:
- `3` → `0011`
- `A` (=10) → `1010`
- Result: `00111010`

## Common mistakes

- ❌ Confusing **RAM** (volatile, loses data when powered off) with **ROM** (non-volatile, retains data).
- ❌ Saying virtual memory is "extra RAM" – it is part of secondary storage *used like* extra RAM.
- ❌ Forgetting to **show all working** in file-size calculations.
- ❌ Mixing up **sample rate** (how often a sample is taken) with **bit depth** (quality of each sample).
- ❌ Saying lossy compression "corrupts" the file – it intentionally removes data to reduce file size; the result is still a usable file.
- ❌ Confusing **resolution** (number of pixels) with **colour depth** (bits per pixel).

## Quick-check questions

1. What does RAM stand for and why is it called volatile?
2. State **two** differences between RAM and ROM.
3. Convert `10011100` from binary to denary.
4. Calculate the file size in bytes of a 320 × 240 pixel image with a 16-bit colour depth.
5. Give **one** advantage and **one** disadvantage of lossy compression compared to lossless compression.

<details>
<summary>Answers</summary>

1. Random Access Memory. It is volatile because its contents are lost when the power is switched off.
2. Any two: RAM is volatile / ROM is non-volatile; RAM holds currently running programs / ROM holds the boot program (BIOS); RAM contents change / ROM contents are fixed; RAM is larger / ROM is smaller.
3. 128+16+8+4 = 156.
4. 320 × 240 × 16 = 1,228,800 bits ÷ 8 = 153,600 bytes.
5. Advantage: smaller file size (1); Disadvantage: some data is permanently lost so quality is reduced (1).

</details>
