# 2.5 Programming Languages and Integrated Development Environments

> **Component:** J277/02 – Computational Thinking, Algorithms and Programming | **Section:** 2.5 | **Exam:** J277/02

---

## What you need to know

Programs can be written in different types of language, from low-level machine code to high-level Python or Java. You need to understand the levels of language, the translators that convert code into machine-readable form, and the tools that IDEs provide to help programmers write, test, and debug code.

## Key terms

| Term | Definition |
|------|-----------|
| Machine code | Binary instructions directly understood by the CPU (0s and 1s) |
| Assembly language | Low-level language using short mnemonics (e.g. ADD, MOV, LDA); one-to-one with machine code |
| High-level language | Human-readable programming language (e.g. Python, Java) that is independent of the hardware |
| Translator | Software that converts source code into machine code |
| Compiler | Translates the whole program into machine code at once, before it runs |
| Interpreter | Translates and executes source code line by line |
| Assembler | Translates assembly language into machine code |
| IDE | Integrated Development Environment – software that provides tools to write, test, and debug programs |
| Source code | The original human-written code in a programming language |
| Object code | The machine code output produced by a compiler |
| Syntax highlighting | Colour-coding different parts of code (keywords, strings, variables) to improve readability |
| Breakpoint | A marker in code where the program pauses during debugging, allowing the programmer to inspect values |
| Auto-completion | IDE feature that suggests or completes code as the programmer types |

## Core notes

### Levels of programming language

**Machine code**
- The only language the CPU understands directly.
- Made up of binary (0s and 1s).
- Very difficult for humans to read or write.
- No translation needed — runs directly on hardware.
- Example: `10110000 01100001`

**Assembly language**
- Uses short, human-readable mnemonics instead of binary.
- Each mnemonic maps to one machine code instruction (one-to-one correspondence).
- Still hardware-specific (different CPU architectures have different assembly languages).
- Requires an **assembler** to convert to machine code.
- Example: `MOV AL, 61h` (move the value 61 hex into register AL)

**High-level language**
- Human-readable, closer to English.
- One statement can translate to many machine code instructions.
- Hardware-independent (the same code can run on different hardware).
- Examples: Python, Java, C#, JavaScript.
- Requires a **compiler** or **interpreter** to translate.

**Comparison:**

| | Machine code | Assembly language | High-level language |
|--|-------------|------------------|---------------------|
| Readable by humans? | No | Somewhat | Yes |
| Hardware-specific? | Yes | Yes | No |
| Translator needed? | No | Assembler | Compiler / Interpreter |
| Development speed | Very slow | Slow | Fast |
| Control over hardware | Full | Full | Limited |
| Typical use | N/A | Device drivers, embedded firmware | Most applications |

### Translators

**Compiler**
- Translates the **entire program** from source code to machine code **before** execution.
- Produces an **object code** file that can be run directly without the compiler.
- Errors are reported all at once after compilation.
- Once compiled, the program runs faster (no translation at run time).
- Distributing compiled programs protects the source code.

**Interpreter**
- Translates and **executes** source code **line by line** at run time.
- If an error is found, the program stops at that line.
- Easier to debug during development (stops immediately at an error).
- Slower at run time because it translates each time the program runs.
- The original source code must be present to run the program.

**Assembler**
- Translates **assembly language** into machine code.
- Simple one-to-one mapping (each mnemonic → one machine code instruction).

**Compiler vs Interpreter:**

| | Compiler | Interpreter |
|--|----------|------------|
| Translates | Whole program at once | One line at a time |
| Output | Object code (machine code file) | No separate output file |
| Errors reported | All at once after full compilation | One at a time, program stops |
| Execution speed | Faster (no translation at run time) | Slower (translates every run) |
| Distribution | Can distribute without source code | Source code required to run |
| Good for | Finished/released programs | Development, scripting, REPL |
| Examples of languages using it | C, C++, Java (JVM) | Python, JavaScript (in browser) |

### Integrated Development Environments (IDEs)

An IDE is a software application that provides a complete environment for writing, testing, and debugging programs.

**Key IDE features:**

| Feature | Description |
|---------|-------------|
| **Code editor** | A text editor designed for writing code with helpful features |
| **Syntax highlighting** | Colour-codes keywords, strings, and variables to improve readability and spot errors |
| **Auto-completion** | Suggests or completes code as the programmer types, reducing errors and saving time |
| **Error diagnostics** | Highlights syntax errors in real time, often with descriptions of the error |
| **Debugger / breakpoints** | Allows the programmer to pause execution at a specific line and inspect variable values |
| **Run-time environment** | Allows programs to be run directly within the IDE |
| **Translator** | The IDE includes or integrates a compiler or interpreter |
| **Line numbering** | Displays line numbers to help locate errors |

## Worked examples

**Example 1 – Compiler vs Interpreter decision**

A company is releasing a finished mobile app. Should they use a compiled or interpreted language?

- **Compiled** is better: faster execution; source code is not exposed to users; no translator needed on the user's device.

A student is learning to program. Should they use a compiled or interpreted language?

- **Interpreted** is better: errors are found immediately line by line; easier to experiment and test code interactively.

---

**Example 2 – IDE tools in practice**

A programmer writes the following code in Python (interpreted):

```python
for i in range(1, 11)
    print(i)
```

The IDE:
1. **Syntax highlighting** – colours the `for` keyword and `print` differently.
2. **Error diagnostics** – underlines `range(1, 11)` with a red squiggle (missing `:` after the `for` statement).
3. **Auto-completion** – suggests `print()` when the programmer starts typing `pr`.

## Common mistakes

- ❌ Saying a compiler is "better than" an interpreter (or vice versa) without context — each has different strengths; the best choice depends on the situation.
- ❌ Saying the compiler "finds all errors" — compilers find syntax errors; logic errors only appear at run time through testing.
- ❌ Confusing assembly language with machine code — assembly uses mnemonics and requires an assembler; machine code is pure binary.
- ❌ Saying high-level languages are "faster" than low-level — low-level code runs closer to the hardware and can be faster to execute, though high-level is much faster to develop.
- ❌ Thinking IDE = programming language — the IDE is the tool used to write code; the language and the IDE are separate things.

## Quick-check questions

1. What is the difference between machine code and assembly language?
2. What does an interpreter do?
3. Give **two** advantages of using a compiler rather than an interpreter for a released program.
4. Give **two** features of an IDE and explain how each helps a programmer.
5. Why are high-level languages more commonly used for application development than low-level languages?

<details>
<summary>Answers</summary>

1. Machine code consists of binary (0s and 1s) directly understood by the CPU; assembly language uses human-readable mnemonics (e.g. ADD, MOV) that are translated to machine code by an assembler. Both are hardware-specific.
2. An interpreter translates source code into machine code and executes it one line at a time; if it finds an error, it stops at that line.
3. Any two: faster execution (no translation at run time); source code not exposed to end users; no translator needed on the user's device.
4. Any two with explanation, e.g. Syntax highlighting — colours different parts of code (keywords, strings) to help the programmer read code and spot errors more easily; Debugger/breakpoints — allows the programmer to pause the program at a specific line and inspect variable values to find logic errors.
5. High-level languages are closer to human language, so programs are faster to write, easier to read and maintain, and can run on different hardware without modification.

</details>
