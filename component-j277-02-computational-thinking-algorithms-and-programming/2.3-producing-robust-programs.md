# 2.3 Producing Robust Programs

> **Component:** J277/02 – Computational Thinking, Algorithms and Programming | **Section:** 2.3 | **Exam:** J277/02

---

## What you need to know

A **robust program** is one that works correctly, handles unexpected input gracefully, and is reliable in use. This topic covers the techniques programmers use to make their code more robust: defensive design, input validation, authentication, and thorough testing. You also need to understand different types of programming error.

## Key terms

| Term | Definition |
|------|-----------|
| Robust program | A program that handles a wide range of inputs correctly, including unexpected or invalid ones |
| Defensive design | Designing a program to anticipate and handle incorrect or unexpected use |
| Input validation | Checking that data entered by a user meets expected criteria before processing it |
| Authentication | Verifying the identity of a user (e.g. username and password check) |
| Syntax error | An error where the code breaks the rules of the programming language (will prevent the program running) |
| Logic error | An error where the code runs but produces an incorrect result (the program does the wrong thing) |
| Test data | Data chosen to test a program and check it behaves correctly |
| Normal data | Test data that is typical/expected input the program should handle correctly |
| Boundary data | Test data at the edge of what is acceptable (just inside and just outside the limits) |
| Erroneous data | Test data that is invalid or outside the accepted range (the program should reject it) |
| Iterative testing | Testing carried out throughout development (testing as you build) |
| Final/terminal testing | Testing carried out at the end of development on the completed program |

## Core notes

### Defensive design

Defensive design means planning for things to go wrong. It includes:
- **Input validation** – check all user input before using it.
- **Authentication** – verify who is using the system.
- **Anticipating misuse** – assume users will enter unexpected values.
- **Error handling** – provide meaningful error messages.
- **Sanitising input** – clean or encode input before processing (especially for databases — prevents SQL injection).

### Input validation

Validation checks that input meets the required rules — it does NOT check whether input is accurate/true, only whether it is the right type/format/range.

| Validation type | Description | Example |
|----------------|-------------|---------|
| **Range check** | Is the value within an acceptable range? | Age must be between 0 and 120 |
| **Type check** | Is the value the correct data type? | Must be an integer |
| **Length check** | Is the value the right length? | Password must be 8–20 characters |
| **Presence check** | Has any value been entered at all? | Username field must not be empty |
| **Format check** | Does the value match a required pattern? | Email must contain `@` |
| **Lookup check** | Is the value from a pre-approved list? | Exam grade must be A*–U |

**Example in OCR ERL:**
```
age = int(input("Enter your age: "))
while age < 0 OR age > 120
    print("Invalid age. Please enter a value between 0 and 120.")
    age = int(input("Enter your age: "))
endwhile
```

### Authentication

Authentication verifies that the user is who they claim to be.

Common methods:
- **Username and password** – most common; must be stored securely (hashed, not plain text).
- **Multi-factor authentication (MFA)** – requires a second proof (e.g. SMS code, authenticator app).
- **Biometrics** – fingerprint, face recognition.

**Simple password check in OCR ERL:**
```
const CORRECT_PASSWORD = "securePass123"
attempts = 0
authenticated = false
while attempts < 3 AND authenticated == false
    password = input("Enter password: ")
    if password == CORRECT_PASSWORD then
        authenticated = true
        print("Access granted.")
    else
        attempts = attempts + 1
        print("Incorrect. " + str(3 - attempts) + " attempt(s) remaining.")
    endif
endwhile
if authenticated == false then
    print("Account locked.")
endif
```

### Testing

**Why test?**
- To check the program works correctly for all expected inputs.
- To find and fix errors before the program is used.
- To ensure the program meets the original requirements.

**Types of testing:**

| Type | When | How |
|------|------|-----|
| **Iterative testing** | Throughout development | Test each component as it is built; fix errors immediately |
| **Final / terminal testing** | At the end of development | Test the completed program against all requirements |

**Test data types:**

| Type | Description | Example (age input, valid range 0–120) |
|------|-------------|---------------------------------------|
| **Normal** | Typical valid input the program should accept | `25`, `67`, `110` |
| **Boundary** | At the exact edge of the valid range (just inside and just outside) | `0`, `120` (just inside); `-1`, `121` (just outside) |
| **Erroneous** | Completely invalid data the program should reject | `"hello"`, `-50`, `200` |

### Errors

**Syntax errors**
- Break the rules of the programming language.
- Caught by the translator (compiler/interpreter) before the program runs.
- Examples: missing brackets, misspelled keyword, incorrect indentation (in Python).
- The program will **not run** until syntax errors are fixed.

**Logic errors**
- The program runs without crashing but produces wrong results.
- NOT caught by the translator — only found through testing.
- Examples: using `>` instead of `>=`; wrong formula; off-by-one error in a loop.

| | Syntax error | Logic error |
|--|-------------|------------|
| Program runs? | No | Yes |
| Detected by translator? | Yes | No |
| Found by? | Compiler/interpreter | Testing / debugging |

## Worked examples

**Example 1 – Choosing test data**

A program asks the user to enter a number of items to order, which must be between 1 and 50.

| Test data type | Value(s) to test | Expected result |
|---------------|-----------------|----------------|
| Normal | `10`, `25`, `40` | Accepted and processed |
| Boundary | `1`, `50` (inside); `0`, `51` (outside) | 1 and 50 accepted; 0 and 51 rejected |
| Erroneous | `-5`, `"hello"`, `999` | Rejected with error message |

---

**Example 2 – Spotting a logic error**

```
// This should print even numbers from 2 to 10
for i = 1 to 10
    if i MOD 2 = 1 then     ← LOGIC ERROR: MOD 2 = 1 gives odd numbers
        print(i)
    endif
next i
```

**Fix:** change `MOD 2 = 1` to `MOD 2 == 0`

## Common mistakes

- ❌ Confusing **validation** with **verification** – validation checks data meets rules; verification checks data was entered correctly (e.g. typing a password twice).
- ❌ Saying a syntax error "gives the wrong answer" – a syntax error prevents the program from running at all.
- ❌ Only including normal test data – always include boundary and erroneous data.
- ❌ Saying boundary data is just "values at the edge" without specifying both inside and outside the boundary.
- ❌ Forgetting that testing continues iteratively throughout development, not just at the end.

## Quick-check questions

1. What is the difference between a syntax error and a logic error?
2. A program accepts exam scores between 0 and 100. Give **one** example of boundary test data and **one** example of erroneous test data.
3. What is input validation?
4. Describe **two** techniques used in defensive design.
5. What is the difference between iterative testing and final testing?

<details>
<summary>Answers</summary>

1. A syntax error breaks the rules of the programming language and prevents the program from running; a logic error allows the program to run but causes it to produce incorrect results.
2. Boundary: `0` or `100` (valid edge) / `-1` or `101` (invalid edge). Erroneous: `"hello"`, `-50`, `200` — any clearly invalid value.
3. Input validation is checking that data entered by a user meets required rules (correct type, range, format, length, etc.) before it is processed.
4. Any two: input validation; authentication; anticipating misuse; providing error messages; sanitising input; using constants to prevent accidental changes to key values.
5. Iterative testing happens throughout development, testing components as they are built; final (terminal) testing happens at the end of development to test the completed program as a whole.

</details>

```
score = int(input("Enter score: ")
if score > 50 then
    print("Pass")
else
    print("Fail")
endif
```

*[Mark scheme: Syntax error: missing closing bracket on `input("Enter score: ")` (1); this will prevent the program from running (1). Logic error: `score > 50` should be `score >= 50` (1) — a score of exactly 50 would incorrectly be output as "Fail" (1). Accept other valid identifications.]*
