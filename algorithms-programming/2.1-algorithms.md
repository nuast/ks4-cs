# 2.1 Algorithms

> **Component:** J277/02 – Computational Thinking, Algorithms and Programming | **Section:** 2.1 | **Exam:** J277/02

## What you need to know

Algorithms are step-by-step instructions for solving a problem. This topic covers the three pillars of **computational thinking**, how to design and represent algorithms (pseudocode and flowcharts), and two key searching and sorting algorithms you must be able to trace, compare, and write.

## Key terms

| Term | Definition |
|------|-----------|
| Algorithm | A precise, ordered set of instructions to solve a problem |
| Computational thinking | A problem-solving approach using decomposition, abstraction, and algorithmic thinking |
| Decomposition | Breaking a complex problem down into smaller, more manageable sub-problems |
| Abstraction | Removing/hiding unnecessary detail so you can focus on what matters |
| Algorithmic thinking | Identifying patterns and designing a set of rules/steps to solve a problem |
| Pseudocode | An informal, language-independent way of representing an algorithm |
| Flowchart | A diagram representing an algorithm using standard shapes |
| Linear search | Checking each element of a list one by one until the target is found or the list is exhausted |
| Binary search | Repeatedly halving a **sorted** list to find the target value |
| Bubble sort | Repeatedly comparing and swapping adjacent elements until the list is sorted |
| Merge sort | Splitting a list in half recursively, then merging sorted halves back together |
| Insertion sort | Building a sorted list one element at a time by inserting each into its correct position |
| Efficiency | How well an algorithm performs in terms of time and memory used |

## Core notes

### Computational thinking

**Decomposition**
- Break a big problem into smaller parts.
- Example: building a game → separate sub-problems: display graphics, handle user input, track score, play sound.

**Abstraction**
- Remove irrelevant detail.
- Example: a map shows roads but not every blade of grass; a program stores a player's name and score but not their hair colour.

**Algorithmic thinking**
- Identify patterns in a problem.
- Design repeatable, generalised rules that solve it.

### Searching algorithms

**Linear (sequential) search**
- Start at the first element; compare it with the target.
- If it matches, return the position.
- If not, move to the next element.
- Repeat until found or the end of the list is reached.
- Works on **unsorted** or **sorted** lists.
- Slow for large lists: must check every element in the worst case.

**Binary search**
- The list **must be sorted** first.
- Find the middle element. Compare with the target.
  - If equal: found!
  - If target < middle: search the left half.
  - If target > middle: search the right half.
- Repeat until found or the search space is empty.
- Much faster than linear search for large sorted lists.

**Comparison:**

| | Linear search | Binary search |
|--|---------------|--------------|
| Requires sorted list? | No | Yes |
| Speed (small list) | Acceptable | Acceptable |
| Speed (large list) | Slow | Fast |
| Simple to implement? | Yes | More complex |

---

### Sorting algorithms

**Bubble sort**
- Compare pairs of adjacent elements; swap if out of order.
- Repeat passes until no swaps occur (list is sorted).
- Easy to understand but slow for large lists.

**Merge sort**
- Split the list in half repeatedly until each sub-list has one element.
- Merge pairs of sub-lists in sorted order.
- Efficient for large lists.

**Insertion sort**
- Take each element and insert it into its correct position in the already-sorted portion.
- Efficient for nearly-sorted lists.

**Comparison:**

| Algorithm | Efficiency | Best use case |
|-----------|-----------|---------------|
| Bubble sort | Slow (poor for large lists) | Simple/educational; almost-sorted lists |
| Merge sort | Fast (efficient for large lists) | Large, unsorted data sets |
| Insertion sort | Medium (good for small/nearly-sorted) | Small or nearly-sorted lists |

## Worked examples

**Example 1 – Linear search trace**

List: `[4, 7, 2, 9, 1, 5]` — Target: `9`

| Step | Index | Value | Match? |
|------|-------|-------|--------|
| 1 | 0 | 4 | No |
| 2 | 1 | 7 | No |
| 3 | 2 | 2 | No |
| 4 | 3 | 9 | Yes – found at index 3 |

---

**Example 2 – Binary search trace**

Sorted list: `[1, 3, 5, 7, 9, 11, 13]` — Target: `7`

| Step | Left | Right | Mid | Mid value | Action |
|------|------|-------|-----|-----------|--------|
| 1 | 0 | 6 | 3 | 7 | Found! |

Target: `11`

| Step | Left | Right | Mid | Mid value | Action |
|------|------|-------|-----|-----------|--------|
| 1 | 0 | 6 | 3 | 7 | 11 > 7 → search right half |
| 2 | 4 | 6 | 5 | 11 | Found! |

---

**Example 3 – Bubble sort trace**

List: `[5, 3, 8, 1, 4]`

**Pass 1:**
- Compare 5, 3 → swap → `[3, 5, 8, 1, 4]`
- Compare 5, 8 → no swap → `[3, 5, 8, 1, 4]`
- Compare 8, 1 → swap → `[3, 5, 1, 8, 4]`
- Compare 8, 4 → swap → `[3, 5, 1, 4, 8]`

**Pass 2:**
- Compare 3, 5 → no swap
- Compare 5, 1 → swap → `[3, 1, 5, 4, 8]`
- Compare 5, 4 → swap → `[3, 1, 4, 5, 8]`

**Pass 3:**
- Compare 3, 1 → swap → `[1, 3, 4, 5, 8]`
- No further swaps in remaining comparisons

**Pass 4:** No swaps → list is sorted: `[1, 3, 4, 5, 8]`

---

**Example 4 – Pseudocode for bubble sort (OCR ERL)**

```
array nums[5] = [5, 3, 8, 1, 4]
swapped = true
while swapped == true
    swapped = false
    for i = 0 to 3
        if nums[i] > nums[i + 1] then
            temp = nums[i]
            nums[i] = nums[i + 1]
            nums[i + 1] = temp
            swapped = true
        endif
    next i
endwhile
```

## Common mistakes

- ❌ Attempting a binary search on an unsorted list – binary search **only works on sorted data**.
- ❌ Confusing decomposition and abstraction – decomposition is about breaking a problem into parts; abstraction is about removing unnecessary detail.
- ❌ Not updating the left/right pointers correctly in a binary search trace.
- ❌ Stopping bubble sort after one pass – it needs to keep passing until no swaps occur.
- ❌ Saying merge sort is "harder" and therefore worse – merge sort is more complex to code but significantly more efficient for large data sets.

- **Define** – decomposition, abstraction, algorithmic thinking, linear search, binary search, bubble sort, merge sort.
- **Trace** – complete a trace table for a linear search, binary search, or bubble sort (very commonly examined).
- **Write** – write pseudocode (ERL) for a simple algorithm.
- **Refine** – fix a given algorithm that contains an error.
- **Compare** – linear vs binary search; bubble vs merge sort.
- **Describe** – the steps of an algorithm in plain English.

## Quick-check questions

1. What is decomposition?
2. What condition must be met before you can use binary search?
3. Describe the steps of a bubble sort in plain English.
4. A list has 1,024 elements. In the worst case, how many comparisons does a binary search need?
5. Give **one** advantage and **one** disadvantage of merge sort compared to bubble sort.

<details>
<summary>Answers</summary>

1. Decomposition is breaking a complex problem down into smaller, more manageable sub-problems.
2. The list must be sorted (in ascending or descending order) before binary search can be used.
3. Compare adjacent pairs of elements; if they are in the wrong order, swap them. Repeat passes through the list until no swaps occur in a full pass — the list is then sorted.
4. log₂(1024) = 10 comparisons in the worst case.
5. Advantage: merge sort is much more efficient (faster) for large data sets; Disadvantage: merge sort is more complex to implement and requires more memory during the merging process.

</details>
